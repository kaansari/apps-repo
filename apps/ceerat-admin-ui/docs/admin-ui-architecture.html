Ceerat Admin UI Architecture

This is a plain-English explanation of apps/ceerat-admin-ui. It explains what the app does, how the folders fit together, and how a browser click becomes a backend admin action.

What it is

A small Go web application for admins to manage users, roles, permissions, and the RBAC cache.

Where it runs

Normally on http://localhost:3010.

Who may use it

Only logged-in users whose role is admin.

What it talks to

The user service gRPC API on localhost:50051 and admin HTTP API on localhost:8081.

Big Picture

The admin UI is not the database owner. It is a browser-facing control panel. It logs an admin in, stores the JWT in an HttpOnly cookie, and forwards admin operations to the backend.

The Internal Parts

main.go

Think of this as the app starter. It says: load settings, connect to backend APIs, prepare the web pages, then listen on the admin port.

starts HTTP server

internal/config

This reads values from environment variables. If a value is missing, it uses local defaults.

  • CEERAT_ADMIN_UI_PORT, default 3010
  • CEERAT_API_BASE_URL, default localhost:50051
  • CEERAT_ADMIN_API_BASE_URL, default http://localhost:8081

internal/apiclient

This is the messenger. It logs in through gRPC, validates the session through the admin API, and sends admin requests with the JWT bearer token.

gRPC + HTTP client

internal/server

This is the admin UI controller. It decides what URL does what, renders templates, sets the session cookie, and blocks non-admins.

routes + sessions

Why The Admin Proxy Exists

The admin proxy is the route POST /api/admin/proxy inside ceerat-admin-ui. It is a safe middleman between browser JavaScript and the real backend admin API.

ReasonPlain-English explanation
Keeps the JWT saferThe admin JWT is stored in an HttpOnly cookie. Browser JavaScript cannot read that cookie directly. The Go server reads it, checks the session, and attaches it to backend calls as Authorization: Bearer <token>.
Keeps browser calls same-originThe browser talks to http://localhost:3010, the admin UI. It does not need to call http://localhost:8081 directly or handle CORS details.
Central admin checkBefore forwarding anything, the proxy calls the same admin session check used by protected admin pages. Non-admin users are rejected before reaching backend admin endpoints.
Limits what can be proxiedThe proxy only accepts backend paths that start with /api/admin/. That keeps it from becoming a general-purpose open proxy.
Simpler frontend codeadmin.js can call one local endpoint and does not need to know backend service addresses, bearer-token headers, or backend auth details.

Short version: the admin proxy lets the browser ask for admin actions while the Go admin UI server handles the sensitive token and admin-only forwarding.

Why Services Use Different Ports

Each port belongs to a different process with a different job. In local development this makes the system easier to start, stop, debug, and secure.

ServiceLocal addressJobWhy separate?
Web UIhttp://localhost:3000Main customer/agent browser experience.Serves the normal application screens separately from the admin tool.
Admin UIhttp://localhost:3010Admin-only browser experience.Can be protected, restarted, or later deployed differently from the public app.
Admin APIhttp://localhost:8081Backend admin endpoints for users, RBAC, and cache refresh.Belongs to the user service, where the data and authorization rules live.
User service gRPClocalhost:50051Core backend API for auth, users, customers, services, and orders.gRPC is a backend service protocol and is kept separate from browser-facing HTML apps.
Agent servicehttp://localhost:8088AI orchestration service that talks to OpenAI and Ceerat gRPC APIs.It can evolve, restart, scale, and handle OpenAI/tool-calling logic without mixing that logic into the web UI or user service.
Postgreslocalhost:55434Database.Only backend services should talk to it directly.

In production, these do not have to be exposed as separate public ports. A reverse proxy could present a cleaner public shape like https://app.ceerat.com/admin while internally routing to ceerat-admin-ui:3010, ceerat-user-service:8081, and ceerat-user-service:50051.

Why The Agent Service Is Separate

The agent service is separate mostly because it has a very different responsibility from the web UI and user service. It is not just a page server and it is not the system of record.

Talks to OpenAI

It owns model calls, tool-calling, AI prompts, and AI-specific safety checks.

Calls platform APIs

It does not write the database directly. It performs work through Ceerat gRPC APIs using the user JWT.

Can evolve independently

New tools, memory, business intelligence, recommendations, and retry behavior can be added without bloating the web UI.

Can restart independently

If AI logic changes or fails, the main web UI and user service can remain separate and easier to reason about.

How To Think About It

Templates are the pages

The templates are mostly static HTML. They create the visible screens and mark important spots with attributes like data-users or data-role-form.

JavaScript fills the pages

admin.js fetches data after the page loads, fills tables, and sends changes back to the Go server.

Go protects the pages

internal/server checks the session cookie before showing admin pages or forwarding admin API requests.

The backend owns the data

The admin UI does not write the database directly. It asks the user service admin API to do the real work.
