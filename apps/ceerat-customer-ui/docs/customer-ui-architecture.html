Ceerat Customer UI Architecture

This is a beginner-friendly explanation of apps/ceerat-customer-ui. It explains the browser pages, Go internals, customer self-registration flow, API calls, AI agent forwarding, and PWA/offline pieces.

What it is

The customer-facing web application for login, registration, profile, services, orders, preferences, and AI chat.

Where it runs

Normally on http://localhost:3005.

Who uses it

Customers with role customer, plus authenticated users using the portal experience.

What it talks to

The Ceerat gRPC user service on localhost:50051 and the agent service on http://localhost:8088.

Big Picture

The customer UI is a Go web server plus normal browser assets. The browser talks to this app over HTTP. The app stores the Ceerat JWT in an HttpOnly cookie, then uses that JWT when calling backend gRPC APIs.

The Internal Parts

main.go

Starts the customer portal. It creates the backend client, creates the HTTP server, listens on the configured port, and shuts down when stopped.

starts HTTP server

internal/config

Reads local settings. Important defaults include CEERAT_CUSTOMER_UI_PORT=3005, CEERAT_AGENT_BASE_URL=http://localhost:8088, and CEERAT_API_BASE_URL.

environment config

internal/apiclient

This is the gRPC adapter. The rest of the web app calls simple Go methods like Login, RegisterCustomer, Dashboard, and CreateOrder.

gRPC client wrapper

internal/server

This controls the browser-facing app: pages, API routes, cookies, customer session checks, agent forwarding, security headers, and JSON responses.

routes + sessions

internal/server/logging.go

Writes structured logs for requests and app activity. In development it logs request/response details after redacting passwords, tokens, and secrets.

safe activity logs

Customer Self-Registration

The customer UI is designed so a public user can create a customer account and immediately get a linked customer profile.

RuleWhat it means
Customer role gets one profileA customer-role user should have exactly one linked customer profile via customers.user_id.
Registration uses a dedicated flowThe preferred flow is RegisterCustomer, which creates the user and customer profile together.
Customers do not create other customersCustomer users should use self-profile APIs such as GetMyCustomerProfile and UpdateMyCustomerProfile.
Admin and agent workflows stay intactAdmins and agents can still create/manage multiple customers through the general customer APIs when RBAC allows it.
customers.user_id remains nullableThis lets admin/agent-created customer rows exist without a linked portal user.

Security Model

Session cookie

The JWT is stored in an HttpOnly cookie named ceerat_session. Browser JavaScript does not need to read the token directly.

JWT metadata

When the server calls protected gRPC APIs, internal/apiclient attaches authorization: Bearer <jwt>.

Customer-owned data

Customer flows use self-service APIs such as GetMyCustomerProfile, UpdateMyCustomerProfile, CreateMyOrder, and ListMyOrders.

Safe logging

Development activity logs sanitize keys containing password, token, or secret before writing details.

How To Think About It

Templates are the screens

The template files are the visible pages users load in the browser.

JavaScript makes screens interactive

app.js submits forms, loads dashboard data, updates tables, and talks to same-origin JSON routes.

Go protects the backend

The Go server checks cookies and only calls protected gRPC APIs with the user JWT.

The user service owns data

The customer UI does not write the database directly. It asks the user service to do the real work.
