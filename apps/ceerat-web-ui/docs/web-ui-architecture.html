Ceerat Web UI Architecture

This is a beginner-friendly explanation of apps/ceerat-web-ui. It explains the main authenticated web application, its Go internals, browser templates, same-origin APIs, gRPC backend calls, orders, dashboard, and AI chat forwarding.

What it is

The main authenticated Ceerat web app for login, registration, dashboard, customers, services, orders, preferences, and AI chat.

Where it runs

Normally on http://localhost:3000.

Who uses it

Authenticated Ceerat users. The backend RBAC model decides which gRPC actions each user can perform.

What it talks to

The Ceerat gRPC user service on localhost:50051 and the AI agent service on http://localhost:8088.

Big Picture

The web UI serves browser pages and exposes small JSON endpoints for the browser. The browser stays same-origin: it talks to the web UI, and the web UI talks to backend services.

The Internal Parts

main.go

Starts the web app. It loads settings, creates the gRPC API client, builds the HTTP server, listens on port 3000 by default, and handles shutdown signals.

starts HTTP server

internal/config

Reads environment variables. Important defaults are CEERAT_WEB_UI_PORT=3000, CEERAT_API_BASE_URL=http://localhost:8080, and CEERAT_AGENT_BASE_URL=http://localhost:8088.

environment config

internal/apiclient

This is the gRPC adapter. The server calls simple Go methods like Login, Dashboard, CreateCustomer, and CreateOrder; the client translates them into gRPC calls.

gRPC client wrapper

internal/server

This controls browser-facing behavior: pages, JSON routes, cookies, auth checks, agent forwarding, security headers, and response formatting.

routes + sessions

internal/server/logging.go

Writes structured logs for HTTP requests and app activity. In development it includes redacted request/response payloads.

safe activity logs

Security Model

Same-origin browser API

The browser calls routes on localhost:3000. It does not call gRPC or the agent service directly.

HttpOnly JWT cookie

The JWT is stored in a cookie named ceerat_session. JavaScript does not need to read it directly.

JWT metadata to gRPC

When the Go server calls protected gRPC methods, internal/apiclient attaches authorization: Bearer <jwt>.

Safe logging

Development logs redact keys containing password, token, or secret before writing request/response payloads.

Different local ports mean different processes with different jobs. In production, a reverse proxy can hide this behind cleaner public URLs.

How To Think About It

Templates are the screens

The template files are the visible pages users load in the browser.

JavaScript makes screens interactive

app.js submits forms, loads dashboard and order data, updates the UI, and calls same-origin JSON routes.

Go protects the backend

The Go server checks cookies and only calls protected gRPC APIs with the user JWT.

The user service owns data

The web UI does not write the database directly. It asks the user service to do the real work.
